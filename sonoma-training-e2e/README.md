# Sonoma Training E2E Harness

Standalone black-box E2E harness for the Koru Android field-test build. This project lives outside the production Android app and interacts through device boundaries only: ADB, UI Automator, mock GPS, logcat, foreground-service state, and recorded-session files.

## What It Exercises

- Installs and launches `com.trustableai.koru.debug`.
- Publishes Sonoma Raceway mock GPS at 10Hz through `com.trustableai.koru.simrunner`.
- Drives the native Compose cockpit with UI Automator: `Telemetry + Map`, `Phone IMU + GPS`, `Sonoma`, and three beginner goals.
- Starts/stops a training session, pulls the newest schema v2 recorded session, and validates field-test evidence.

The companion app does not import Koru production code. The scenario fixture copies Sonoma coordinates into this project and is generated by `tools/generate-scenario.mjs`.

## Run

```bash
cd sonoma-training-e2e
./run.sh --scenario sonoma_beginner_training.v1 --package com.trustableai.koru.debug
```

The runner writes `reports/<timestamp>/report.json`, `report.md`, `junit.xml`, `logcat.txt`, `instrumentation.txt`, `recorded-session.json`, and a service dump.

## Local Checks

cd sonoma-training-e2e
npm test
JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home" ../pixel-android-app/gradlew -p . :app:assembleDebug :app:assembleDebugAndroidTest

## Validation Rules

- Native cockpit reached through UI Automator.
- `KoruTelemetryService` visible while active.
- Logcat contains `Telemetry loop started source=phone_imu_gps cameraFusion=false`.
- No fatal exception, ANR, or process-death marker in the captured log window.
- Saved artifact is schema `2`, track is `Sonoma Raceway`, frames include `phone_imu_gps`, and the artifact is fresh for the run.
- Decisions include FEEDFORWARD, HOT, and P0 `BRAKE` with text `Brake now`.
- Audio events link to known decision IDs, and P0 audio has `CLIP_STARTED` or flushed TTS fallback evidence.
- HOT p95 latency is below `50ms`; P0 audio dispatch is below `100ms`.
- LiteRT/MediaPipe validation is conditional. If no model/status is observed, the GPU assertion is reported as skipped.

## Notes

Android does not expose a public mock sensor provider equivalent to mock GPS. The scenario encodes late braking, over-slow/coast, early-throttle, and recovery labels, but pass/fail is anchored to evidence the app can produce from public device boundaries: GPS-derived telemetry, coaching decisions, audio evidence, and logs.
