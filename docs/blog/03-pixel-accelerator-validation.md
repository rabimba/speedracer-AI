# Pixel Accelerator Validation

The Pixel 10 accelerator check is designed to compare CPU, GPU, and NPU behavior without forcing an unsafe runtime path.

The benchmark artifact is generated by:

```bash
cd pixel-android-app
JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home" ./gradlew connectedDebugAndroidTest \
  -Pandroid.testInstrumentationRunnerArguments.class=com.trustableai.koru.runtime.benchmark.AcceleratorComparisonInstrumentedTest#compareCpuGpuNpuTokenGenerationSpeed \
  -Pandroid.testInstrumentationRunnerArguments.runsPerStrategy=1
```

The output file is:

```text
/sdcard/Android/data/com.trustableai.koru.debug/files/accelerator-comparison-report.json
```

## What The Report Measures

The CPU and GPU rows use MediaPipe `LlmInference` token generation. When a compatible model is staged, those rows report median tokens per second, time to first token, and latency.

The NPU row is currently status-only. In this build, AICore is scaffolded as an on-device backend, but no measured Prompt API token-generation lane is implemented yet. The report marks that explicitly instead of inventing an NPU speed.

## Current Pixel 10 Result

On the connected Pixel 10, the model is present and checksum-valid. The native MediaPipe path is still blocked because the staged artifact contains `tf_lite_audio_adapter`, which the bundled runtime does not support.

That means the honest current comparison is:

- CPU token generation: unavailable with the current model/runtime pair.
- GPU token generation: unavailable with the current model/runtime pair.
- NPU token generation: not implemented; AICore is status-only.
- Deterministic coaching: available and tested.

## Next Validation Step

The next accelerator milestone is to stage a compatible text-only LiteRT-LM model or update the Android runtime to one that supports the current artifact. Once that is done, the same benchmark test will produce real CPU and GPU token speeds without changing the report format.
