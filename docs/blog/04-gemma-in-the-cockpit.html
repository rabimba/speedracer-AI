Putting Gemma in the Cockpit

On-device Gemma 4 E2B, LiteRT-LM, and a week lost to a single tensor named tf_lite_audio_adapter — followed by the afternoon the Neural Processing Unit finally beat everything else on the phone.

By Part 3 the architecture had a slow, smart lane fully decoupled from the fast, dumb one. That decoupling was the permission slip for the most ambitious idea in the project: run a real language model on the phone. No cloud, no round-trip, no dependency on a cell signal at a track in the hills — just a 2.3-billion-parameter model reasoning about your driving from the passenger seat, literally.

The model is Gemma 4 E2B — the "2B-effective" variant that punches far above its active-parameter count and is small enough to live on a Pixel. The plan: package it as a .litertlm artifact and run it through the on-device LiteRT-LM runtime for the EDGE lane's tactical enrichment, reserving the cloud Gemini call strictly for big-picture strategy. On-device for what should be private and fast; cloud only for what can afford a round trip.

Because of that: the native rewrite

The earliest builds were a hybrid — a WebView wrapping a web app. Fine for a demo, wrong for a field tool. To actually own the camera, foreground services, local audio, telemetry fusion, and model lifecycle, the app went fully native: Kotlin + Jetpack Compose + Material 3, with a clean reactive flow (Compose UI → LiveSessionViewModel → KoruSessionBus StateFlow). Killing the WebView removed a class of serialization and state-sync bugs and put the hot path right next to the Android sensors.

I built the benchmark harness to want real numbers, not vibes: a warmup pass to discount cold-start, time-to-first-token captured off the streaming callback, and median/p95 tokens-per-second and latency across runs, per accelerator. Then I went to load the model. And the model said no.

Until one day: a single rejected tensor

The native runtime refused to load the Gemma 4 E2B .litertlm for token generation, because it rejects an operation called tf_lite_audio_adapter baked into that artifact's graph. The export I had was multimodal, carrying audio-adapter machinery the text-generation runtime won't accept. So modelReadyForNativeAndroid stayed stubbornly false, and the harness did exactly what I designed it to do: it refused to report a number it couldn't honestly measure. For a while, every lane in the chart was a dashed "PENDING" box — a measured absence, not a fake bar.

Because of that: a text-only export and a Tensor-shaped key

Unblocking it took three specific moves, and the NPU needed all three:

  • A text-only re-export. Re-exporting Gemma 4 E2B without the audio adapter produced a graph the runtime accepts. modelReadyForNativeAndroid flipped to true and the GPU and CPU lanes lit up immediately.
  • A Tensor-specific artifact for the NPU. The NPU doesn't run the generic file — it runs a dedicated gemma-4-E2B-it_Google_Tensor_G5.litertlm, compiled for the Pixel's Tensor G5, dispatched through the packaged libLiteRtDispatch_GoogleTensor.so runtime in the APK. No dispatch library, no NPU.
  • Per-lane process isolation. A native NPU dispatch failure can abort the whole process, which would have taken the other lanes down with it. So the runner now runs each accelerator in its own instrumentation process and merges the per-lane reports afterward. A crash on one lane is recorded as that lane's result, not a lost run.

Until finally: the numbers

With a native-ready model and the Tensor dispatch runtime in place, all three lanes generate tokens through the official LiteRT-LM benchmark API. Here is the real chart the pipeline produces — no longer a wall of PENDING:

LaneRuntime Median tok/sTTFT
GPULiteRT-LM10.31108 ms
CPULiteRT-LM6.98545 ms
NPULiteRT-LM · Tensor G514.3424 ms

The NPU isn't just the throughput winner at 14.3 tok/s (about 2× the CPU). The number that actually matters for a coach is time-to-first-token: 424 ms on the NPU, versus 1.1 s on the GPU and a brutal 8.5 s on the CPU. TTFT is how long the driver waits before the first word of enrichment arrives — and on the CPU that wait is long enough that the moment is gone. The NPU's sub-half-second first token is the difference between enrichment that feels present and enrichment that feels like a memory. That's why the efficient silicon matters here specifically: not for peak throughput bragging rights, but for latency-to-first-word.

One honest footnote on "the NPU"

Worth being precise, because trustability is the whole point. The measured NPU lane above runs Gemma through LiteRT-LM on the Tensor dispatch backend — that's what produced 14.3 tok/s. Separately, the in-app conversational reasoner (AiCoreReasoner) still treats Google's AICore Prompt API as a health-probed lane rather than its primary generator:

// AiCoreReasoner.warmup() — a health probe for the AICore Prompt API lane.
usable = runCatching { Class.forName("com.google.ai.edge.aicore.GenerativeModel") }.isSuccess
val state = if (usable) DEGRADED else UNAVAILABLE

So: the benchmark proves the Tensor NPU can generate Gemma tokens fast, and it's now the default EDGE-lane target. The conversational AICore Prompt API integration is still a wired, observable lane we can promote when its structured-output path is production-ready. Both things are true at once, and saying so plainly is the point — trustability cuts both ways: the app is honest with the driver, and the writeup is honest with you.

The decoupling from Part 3 is what made this safe to chase: the model lane could be blocked, then slow, then fast — and the car kept getting coached the whole time.

The smart lane finally generates tokens, on the most efficient silicon in the phone, in under half a second to first word. Which leaves the question the whole series has been circling: deterministic core, adaptive model, honest numbers — how do you put it together and actually trust it on a track? Next, the finale: earning trust at speed — determinism, on-device validation, and the recorded Pixel session that proves the core is real.
