## Architecture, as built

The "Split-Brain" engine routes every telemetry frame through three asynchronous
lanes against a 300–500 ms end-to-end budget, on the principle that **"feedback
800 ms late is worse than silence."**

- **HOT (<50 ms)** — a deterministic `DECISION_MATRIX` of hand-written, auditable
  heuristics. Fires P0 safety ("Brake!") and P1 tactical advice with no network and
  no LLM. This is the lane that was measured at 5 ms p95 on device.
- **COLD (2–5 s)** — an async call to **Gemini 2.5 Flash Lite** in the cloud for
  "why, not what" strategic analysis. The loop never waits on it.
- **FEEDFORWARD (geofenced)** — GPS-triggered, corner-specific advice ~150 m out,
  now velocity-scaled.

A `DriverModel` classifies the driver as BEGINNER / INTERMEDIATE / ADVANCED on a
10-second rolling window with 5-second hysteresis (input smoothness + coasting
ratio), and that classification modulates coaching language, cadence, session
progression, and the LLM prompt. This is the "coaching paradigm."

## The on-device model situation (read this carefully)

The intended EDGE lane is **Gemma 4 E2B** packaged as `.litertlm`, executed through
**MediaPipe `LlmInference`**. The benchmark harness (`LlmBenchmarkRunner.kt`) is
well-built: warmup pass, per-run TTFT capture via a `ProgressListener`, median/p95
tokens-per-second and latency. It is gated behind `modelReady()`, which requires the
model to be present, checksum-verified, **and** `supportsNativeAndroid`.

That flag started the day **false**: the original Gemma 4 E2B `.litertlm` was a
multimodal export rejected for native token generation because the runtime rejects
`tf_lite_audio_adapter`. The harness correctly refused to fake numbers and reported
PENDING.

### NPU — now a measured lane (and a separate status probe)

Two distinct things, kept distinct on purpose:

- **NPU token generation (measured):** runs Gemma through **LiteRT-LM on the Google
  Tensor dispatch backend** (`libLiteRtDispatch_GoogleTensor.so` packaged in the APK)
  with a Tensor-G5 artifact `gemma-4-E2B-it_Google_Tensor_G5.litertlm`. This is the
  14.3 tok/s / 424 ms TTFT lane above — the fastest on the phone, and the lowest
  time-to-first-token by a wide margin.
- **`AiCoreReasoner` (status probe):** still a reflection check for the AICore Prompt
  API (`Class.forName("com.google.ai.edge.aicore.GenerativeModel")`) reporting
  `DEGRADED`/`UNAVAILABLE`. This is a separate, observable lane — not the measured
  generator — and remains intentionally labeled as such.

  `LlmBenchmarkInstrumentedTest`, `AcceleratorComparisonInstrumentedTest`,
  `SonomaTrainingE2ETest`. Require a connected device/emulator.
- **Web app unit tests** (`koru-application`, vitest): the checked-out `node_modules`
  carries only the macOS rolldown native binding; this Linux/arm64 sandbox needs
  `@rolldown/binding-linux-arm64-gnu`, and the npm registry is not reachable here. On
  the target machine a clean `npm ci` resolves this. **This is an environment quirk,
  not a code defect.**

## Fixes applied in this pass

1. **Decoupled the NMEA parser from the server stack.** `ingest.py` previously
   imported FastAPI, sse-starlette and pyserial-asyncio at module top, so the "unit"
   test could not run without the entire server installed. The pure parser now lives
   in `streaming-telemetry-server/nmea.py` (stdlib + `pynmea2` only); `ingest.py` and
   the test import from there. Verified: `import nmea` pulls in neither FastAPI nor
   serial, and the 4 tests pass.
